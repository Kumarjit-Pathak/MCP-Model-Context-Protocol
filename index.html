What is MCP?

Model Context Protocol (MCP) is an open standard that defines how AI agents communicate with external systems. It defines:

üîç

Discovery

How agents find available tools

‚ö°

Invocation

How agents call those tools

üìã

"MCP is an open protocol that standardizes how applications provide context to large language models (LLMs). It simplifies how models access tools, data, and workflows‚Äîacting as a universal interface layer."

‚Äî Anthropic

MCP Architecture and Core Concepts

The Three Key Components

üß† MCP Client (Agent)

Your AI agent‚Äîthe chai-loving decision-maker. The client:

  • Discovers available tools from servers
  • Decides which tool to invoke based on user goals
  • Sends structured requests to servers
  • Processes responses and continues workflows
Examples: Claude Desktop, Cursor IDE, custom LLMs

‚öôÔ∏è MCP Server (Tools)

A server that exposes:

  • Tools: Functions the agent can call
  • Resources: Data the agent can access
  • Prompts: Pre-defined workflows
Examples: Weather API server, Database query server, Payment server

üîå MCP Protocol (Standard)

The protocol defines:

  • Discovery API: GET /discover
  • Invocation API: POST /invoke/{toolName}
  • Response Format: Structured JSON
  • Security Layer: Auth, authorization, rate limiting

Step 5: Build the Intelligent Client

Now for the fun part: building a client that actually uses an LLM to decide which tools to call. This is where agentic behavior comes to life.

"""
MCP Client - The brains of the operation
"""
import asyncio
from typing import Dict, Any, List
from mcp import ClientSession, StdioServerParameters
from mcp.client.stdio import stdio_client
from anthropic import Anthropic  # or use OpenAI

class MCPClient:
    def __init__(self):
        self.tools: List[Dict[str, Any]] = []
        self.conversation_history: List[Dict[str, Any]] = []
        self.session = None
        self.llm = Anthropic(api_key=os.getenv("ANTHROPIC_API_KEY"))

    async def connect_to_server(self, server_script: str):
        """Connect to an MCP server via stdio."""
        server = StdioServerParameters(
            command="python",
            args=[server_script],
            env=None
        )

        # Create stdio client
        stdio = stdio_client(server)
        read_stream, write_stream = await stdio.__aenter__()
        self._stdio_context = stdio

        # Create and initialize session
        self.session = ClientSession(read_stream, write_stream)
        await self.session.__aenter__()
        await self.session.initialize()

        # Discover available tools
        tools_result = await self.session.list_tools()
        for tool in tools_result.tools:
            self.tools.append({
                "name": tool.name,
                "description": tool.description,
                "inputSchema": tool.inputSchema
            })

        print(f"Connected! Discovered {len(self.tools)} tools")

    async def chat(self, user_message: str) -> str:
        """Process user message using LLM and tool calling."""
        self.conversation_history.append({
            "role": "user",
            "content": user_message
        })

        # LLM decides which tool to use
        response = self.llm.messages.create(
            model="claude-3-5-sonnet-20241022",
            max_tokens=4096,
            messages=self.conversation_history,
            tools=self.tools
        )

        # If LLM wants to call a tool
        if response.stop_reason == "tool_use":
            for tool_use in response.content:
                if tool_use.type == "tool_use":
                    # Invoke tool via MCP
                    result = await self.session.call_tool(
                        tool_use.name,
                        tool_use.input
                    )

                    # Get LLM's final response
                    # ... (handle tool result and get final answer)

        return response.content[0].text

üß† How It Works

  1. Discovery: Client connects to server and asks "What tools do you have?"
  2. User Query: "What's the weather in London?"
  3. LLM Decision: Claude sees available tools and decides to call get_current_weather
  4. Tool Execution: Client invokes the tool via MCP protocol
  5. Response: LLM formats the result naturally: "It's 15¬∞C and cloudy in London"

This is agentic AI in action‚Äîthe LLM autonomously selects and uses the right tool.

Production Deployment

So you've built your Weather, News, and Finance MCP servers. They work beautifully on your laptop. Now what?

Here's the thing about MCP servers using stdio transport: they're different from your typical REST APIs. They don't listen on ports. They don't have HTTP endpoints. They communicate through standard input/output streams‚Äîlike old-school Unix pipes, but with JSON-RPC superpowers.

Understanding MCP Deployment Models

üîç Two Deployment Patterns

1. Local/Desktop Integration: MCP servers run as local processes spawned by desktop applications (Claude Desktop, Cursor, etc.). The client starts the server as a subprocess and communicates via stdio.

  • Perfect for: Personal productivity tools, IDE integrations, local dev workflows
  • Security: Process isolation, file system permissions
  • Examples: Our Weather/News/Finance servers run this way!

2. HTTP Transport (Alternative): MCP servers can optionally expose HTTP endpoints for remote access, though this requires additional setup and security considerations.

  • Perfect for: Centralized enterprise services, cloud-based agents
  • Security: HTTPS, authentication, rate limiting required
  • Examples: Shared company-wide MCP servers

Deployment Option 1: Desktop/Local Applications

The most common pattern‚Äîand what our implementation uses‚Äîis desktop integration:

Security & Best Practices

Building an MCP server is easy. Building a secure, trustworthy, production-grade MCP server? That's where the real work begins. Let's talk about the things that keep security engineers up at night.

Security Model: Stdio vs. HTTP

üìç Good News First

Our Weather/News/Finance servers use stdio transport, which has a built-in security advantage: process isolation. The MCP server runs as a child process of your application, inheriting the same permissions. There's no network exposure, no CORS headaches, no TLS certificates to manage.

Think of it like this: It's more secure to hand someone a sealed envelope (process communication) than to shout across a crowded room (network communication).

üîê

Never hardcode secrets. Use environment variables:

OPENWEATHER_API_KEY=xyz
NEWS_API_KEY=abc

Limitations and Real-World Considerations

Time for some truth serum. FastMCP makes building MCP servers easy, but it doesn't magically solve all problems. Here are the honest limitations you'll face:

1. External API Dependencies

Our Weather, News, and Finance servers depend on third-party APIs. If OpenWeatherMap goes down, your weather tools stop working. If you hit NewsAPI's rate limit, headlines dry up.

Mitigation:

  • Implement caching (reduce API calls by 80%+)
  • Add fallback mechanisms (multiple API providers)
  • Graceful degradation (return cached data when API fails)
  • Monitor API health and quotas

2. Latency & Cumulative Overhead

Every tool call involves: LLM decision (500ms) ‚Üí Tool invocation (200ms) ‚Üí External API (300-1000ms) ‚Üí LLM formatting (500ms). A simple "What's the weather and news in Tokyo?" could take 3-4 seconds.

Reality Check:

User: "What's the weather in London and Tokyo?"
‚îî‚îÄ LLM: Decides to call get_current_weather twice (500ms)
   ‚îú‚îÄ Call 1: London (1200ms total)
   ‚îî‚îÄ Call 2: Tokyo (1200ms total)
‚îî‚îÄ LLM: Formats response (500ms)
Total: ~3.4 seconds

Mitigation: Parallel tool calls (when supported), caching, streaming responses

3. Cost Accumulation

LLM costs add up quickly. Each conversation involves multiple API calls:

  • Initial LLM decision: ~1000 tokens ($0.001-0.01)
  • Tool result processing: ~2000 tokens ($0.002-0.02)
  • Final response: ~500 tokens ($0.0005-0.005)
  • Per conversation: $0.004-0.04

At 1000 conversations/day: $4-40/day or $120-1200/month just in LLM costs.

Mitigation: Use cheaper models (GPT-4o-mini, Claude Haiku), implement conversation limits, cache common queries

4. Stdio Transport Limitations

Our implementation uses stdio, which is perfect for local use but has constraints:

  • One client per server: Can't share a weather server across multiple users simultaneously
  • Local only: Can't access servers remotely without HTTP wrapper
  • Process overhead: Each server is a separate Python process

When this matters: Multi-user enterprise scenarios (solution: switch to HTTP transport)

5. LLM Tool Selection Isn't Perfect

Sometimes the LLM gets confused and calls the wrong tool, or forgets to call a tool at all.

Example failures we've seen:

  • "What's the weather?" ‚Üí Calls get_forecast instead of get_current_weather
  • "Convert 100 USD to EUR" ‚Üí Calls get_exchange_rates and does math wrong
  • "Find news about AI" ‚Üí Returns general news instead of calling search_news

Mitigation: Better tool descriptions, few-shot examples in system prompts, tool-specific validation

6. Complexity Doesn't Disappear‚ÄîIt Shifts

FastMCP eliminates boilerplate, but you still need:

  • Good error handling (network timeouts, API failures)
  • Proper logging and monitoring
  • Dependency management and updates
  • API key rotation and security
  • Rate limiting and caching strategies
  • Testing (unit tests, integration tests, LLM behavior tests)

FastMCP is a power tool, not a magic wand. You're trading "building infrastructure" for "building good tools."
